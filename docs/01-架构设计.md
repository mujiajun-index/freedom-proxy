# 架构设计

本文档描述 freedomProxy 的整体架构、技术选型、模块划分、目录结构、请求处理流程，以及从参考实现 `main.ts` 的演进路径。

## 1. 总体架构

freedomProxy 采用经典的「nginx 反向代理 + Node.js 应用服务」两层结构。nginx 负责对外入口、HTTPS 终止与（可选的）IP 白名单；Node.js 负责代理转发引擎、管理后台与后台 API；配置与映射规则持久化在单个 `config.json` 文件中。

```
                        ┌─────────────────────────────────────────────┐
   客户端 ───HTTPS/HTTP──▶  nginx（对外 80/443）                          │
                        │  · HTTPS 终止                                  │
                        │  · 反向代理到 Node                             │
                        │  · （可选）IP 白名单：allow/deny 或 geo          │
                        │  · 透传 X-Real-IP / X-Forwarded-For            │
                        └──────────────────────┬──────────────────────┘
                                               │ proxy_pass http://127.0.0.1:3000
                                               ▼
                        ┌─────────────────────────────────────────────┐
                        │  Node.js + TypeScript 应用（监听 :3000）       │
                        │                                              │
                        │  请求入口 / 路由分发                           │
                        │      │                                       │
                        │      ├──▶ IP 白名单中间件（app 层兜底）         │
                        │      ├──▶ 分流判断                            │
                        │      │     ├─ /{token}、/{token}/* → 管理后台  │
                        │      │     │     · 静态资源（public/）          │
                        │      │     │     · 后台 API（需鉴权）           │
                        │      │     └─ 命中映射前缀 → 代理引擎转发        │
                        │      │                                       │
                        │  访问日志：logs/access.jsonl                  │
                        │    （IP/地址/时间/命中映射/状态码/耗时）        │
                        │  存储层：config.json（映射 + 配置 + 白名单）     │
                        └─────────────────────────────────────────────┘
                                               │
                                               ▼
                                       上游目标服务（OpenAI / 内网 API / …）
```

## 2. 技术选型理由

| 关注点 | 选型 | 理由 |
| --- | --- | --- |
| 运行时 | Node.js | 生态成熟，nginx 反代最常见，部署资料丰富，对运维友好 |
| 语言 | TypeScript | 类型安全，利于维护代理引擎与配置结构；可直接编译为 JS 运行 |
| 前端 | 原生 HTML/CSS/JS | 后台页面轻量，无需前端框架，便于 nginx 直接托管或 Node 内联托管 |
| 存储 | JSON 文件 | 映射规则量级小，单文件零依赖、可移植、易备份，满足 nginx 部署的无数据库诉求 |
| 入口安全 | 随机 token + 登录 + 白名单 | 三层叠加：地址不可猜（token）、身份需验证（登录）、来源受限（白名单） |

## 3. 模块划分

应用内部按职责划分为以下模块（对应 `src/` 下的子目录）：

- **server（服务与路由分发）**：创建 HTTP 服务，解析请求路径，依次经过白名单中间件后分流到「管理后台」或「代理引擎」。
- **proxy（代理引擎）**：核心转发逻辑。根据请求路径匹配映射前缀，剥离前缀后拼接到目标 URL，透传 method/headers/body，回传响应。其中 `websocket.ts` 处理 `Upgrade` 升级请求：握手透传 + 客户端与目标之间的透明双向裸流（`http://`→`ws://`、`https://`→`wss://`）。
- **auth（鉴权）**：管理员登录校验、密码哈希比对、会话签发与校验（cookie/JWT）。仅管理后台 API 受其保护，代理路由默认不需登录（由白名单/业务方自行控制）。
- **store（存储层）**：`config.json` 的加载、内存缓存、原子写入。提供映射 CRUD、白名单读写、管理员凭据读写等接口。
- **logger（访问日志）**：记录每次请求的 IP、地址、时间、命中映射、状态码、耗时等，追加写入 `logs/access.jsonl`，支持轮转、清理与后台查询。
- **public（管理后台静态资源）**：HTML/CSS/JS，登录页与后台管理页。

## 5. 请求处理流程

一个进入 Node 服务的请求按如下顺序处理：

```
请求进入
   │
   ▼
[1] IP 白名单中间件
   │  读取 config.ipWhitelist；为空 → 放行
   │  非空 → 取客户端 IP（优先 X-Real-IP / X-Forwarded-For，需 TRUST_PROXY）
   │        命中白名单 → 放行；未命中 → 403
   ▼
[2] 路由分流
   │
   ├── 路径以 /{token} 开头？
   │     ├── 是 → 管理后台
   │     │     ├── /{token}/api/login      → 登录（无需会话）
   │     │     ├── /{token}/api/*          → 后台 API（需会话鉴权）
   │     │     └── /{token}、/{token}/...  → 返回 public/ 静态页面
   │     │
   │     └── 否 → 进入代理引擎
   │
   ▼
[3] 代理引擎
   │  遍历启用的 mappings，按「最长前缀」匹配请求路径
   │  未命中 → 返回提示或 404
   │  命中 → 剥离前缀，拼接到 target，构造代理请求：
   │         method = 原 method
   │         headers = 原 headers（剔除与代理自身冲突的项，如 Host）
   │         body    = 原 body 流
   │  fetch 目标 → 回传 status / headers / body
   ▼
[4] 访问日志记录
   │  每个请求（后台或代理）处理完成后记录一条：
   │  time / ip / method / path / status / elapsedMs / mapping / target
   │  按 accessLog 配置写入 logs/access.jsonl（默认记录全部请求）
   ▼
响应返回客户端
```

**关键设计点：**

- **随机 token 隔离后台**：只有路径以 `/{token}` 开头的请求才会进入管理后台逻辑，其它路径（包括根路径 `/`）均不暴露后台入口，降低被扫描风险。
- **最长前缀匹配**：当存在 `/api` 与 `/api/v2` 两条映射时，请求 `/api/v2/x` 优先匹配更具体的 `/api/v2`，避免误转发。
- **token 固定**：token 在首次启动时生成并写入 `config.json`，之后不再变化（详见 [配置说明](03-配置说明.md)），nginx 无需为此特判。
- **全量访问日志**：每个请求（后台或代理）处理完成后记录一条日志，写入独立 JSONL 文件（`logs/access.jsonl`），与 `config.json` 分离，便于轮转与审计。

## 6. 与 main.ts 的演进对比

参考实现 `main.ts` 是一个最小可用的 Deno 代理，本项目在其核心转发思路之上做了产品化扩展：

| 维度 | main.ts（参考） | freedomProxy（本项目） |
| --- | --- | --- |
| 运行时 | Deno | Node.js + TypeScript |
| 代理目标数量 | 单个（一个全局 targetUrl） | 多个（mappings 数组，路径前缀 → 目标） |
| 配置方式 | URL 查询参数 `?setUrl=` | Web 管理后台表单 + 后台 API |
| 存储 | Deno KV | 单个 `config.json` 文件 |
| 鉴权 | 无 | 管理员登录 + 会话 |
| 后台入口 | 固定 `/proxy` | 随机 `/{token}`（首启生成后固定） |
| 访问控制 | 无 | IP 白名单（`;` 分隔 / 单 IP / CIDR） |
| 访问日志 | 控制台打印 | 结构化访问日志（JSONL 文件 + 轮转 + 后台查询） |
| 部署 | Deno 直接运行 | nginx 反代 + Node 服务 |

**保留的核心思路**：剥离路径前缀后拼接到目标 URL、透传 method/headers/body、流式回传响应体——这些在 `main.ts` 的 `/proxy` 分支中已经体现，本项目的代理引擎将其泛化为「按前缀查表 + 转发」。
