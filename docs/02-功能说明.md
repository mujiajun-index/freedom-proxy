## 1. 管理员登录

**目的**：保护管理后台，仅授权管理员可增改映射与白名单。

- **登录入口**：`POST /{token}/api/login`，提交 `username` + `password`。
- **凭据来源**：账号与密码哈希存于 `config.json`（`adminUser` / `adminPasswordHash`），密码也可通过环境变量 `ADMIN_PASSWORD` 覆盖。详见 [配置说明](03-配置说明.md#管理员凭据)。
- **会话**：登录成功后下发会话凭证（HttpOnly Cookie 或 JWT），后续后台 API 请求需携带；会话有有效期，过期需重新登录。
- **失败保护**：登录失败返回统一错误提示，建议对连续失败做简单限流，避免暴力破解。

> 代理转发路由（如 `/openai/*`）**默认不要求登录**——它是给业务客户端调用的。后台访问控制由「随机 token + 登录 + IP 白名单」三重保障。

## 2. 代理映射管理

**目的**：以「路径前缀 → 目标 URL」的方式自由配置代理转发。

### 2.1 映射字段

每条映射包含以下字段：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | string | 是 | 唯一标识（后台生成，如 UUID/时间戳） |
| `prefix` | string | 是 | 本地路径前缀，须以 `/` 开头，如 `/openai` |
| `target` | string | 是 | 目标基础 URL，如 `https://api.openai.com`（一般不带尾斜杠） |
| `enabled` | boolean | 否 | 是否启用，默认 `true`；禁用后该前缀不再转发 |
| `note` | string | 否 | 备注，便于辨识用途 |

### 2.2 匹配与转发规则

- **前缀匹配**：请求路径以某条启用映射的 `prefix` 开头即命中。
- **最长前缀优先**：同时命中多条时，取 `prefix` 最长者（如 `/api/v2` 优先于 `/api`）。
- **路径拼接**：命中后剥离 `prefix`，将剩余路径拼接到 `target` 之后。
- **查询参数透传**：原请求的 query string 原样附加。
- **方法/头/体透传**：`method`、请求头（剔除 `Host` 等冲突项）、请求体原样转发。
- **响应原样回传**：目标响应的 status、headers、body 流式回传给客户端。

### 2.3 用例示例

配置一条映射：

```
prefix : /openai
target : https://api.openai.com
enabled: true
```

请求与转发对照：

```
请求：  GET  /openai/v1/chat/completions?key=xxx
转发：  GET  https://api.openai.com/v1/chat/completions?key=xxx

请求：  POST /openai/v1/embeddings
转发：  POST https://api.openai.com/v1/embeddings
```

> 若 `target` 自带路径（如 `https://gateway.example.com/api`），则 `/openai/v1/x` 会转发到 `https://gateway.example.com/api/v1/x`。建议保持 `target` 仅为 origin（协议+主机），把路径放进 `prefix` 之外的部分，行为最直观。

## 3. 随机管理地址

**目的**：避免后台暴露在固定、可猜测的路径上（类似宝塔面板的安全入口）。

- **生成时机**：首次启动时若 `config.json` 中 `adminToken` 为空，则随机生成一段 token（如 8 位十六进制 `32dfa51e`）并写入配置。
- **固定不变**：生成后**不再变化**，写入 `config.json` 持久化，重启后保持同一地址，便于长期访问。
- **访问方式**：管理后台地址为 `http(s)://你的域名/{token}`，例如 `https://proxy.example.com/32dfa51e`。
- **隐藏性**：根路径 `/` 与其它非映射路径均不暴露后台入口，扫描者难以发现。
- **首次获取**：token 在启动日志/控制台打印一次；若遗忘，可直接查看 `config.json` 中的 `adminToken` 字段。

> 安全建议：生产环境务必配合 HTTPS；如需「换地址」，删除 `config.json` 中的 `adminToken` 后重启即可重新生成（旧地址立即失效）。

## 4. IP 白名单

**目的**：限制只有指定来源 IP 才能访问本服务（含代理路由与管理后台）。

### 4.1 配置格式

- 配置项 `ipWhitelist` 为字符串，**多个条目用 `;` 分隔**。
- 支持两种条目：
  - **单 IP**：如 `192.168.1.10`
  - **CIDR 网段**：如 `10.0.0.0/24`、`2001:db8::/32`（IPv4 / IPv6 均可）
- **为空表示不限制**：当 `ipWhitelist` 为空字符串或未配置时，所有 IP 均可访问。

示例：

```
192.168.1.10;10.0.0.0/24;203.0.113.5
```

含义：允许 `192.168.1.10`、`10.0.0.0/24` 网段、`203.0.113.5` 三个来源；其余一律拒绝（返回 `403`）。

### 4.2 生效层级

IP 白名单可在两处生效（详见 [部署指南](05-部署指南.md#3-ip-白名单落地)）：

- **nginx 层（推荐）**：用 `allow`/`deny` 或 `geo` 在请求到达 Node 前就拦截，效率最高；支持 CIDR。
- **应用层（兜底）**：Node 中间件读取 `X-Real-IP` / `X-Forwarded-For`（需开启 `TRUST_PROXY`）与白名单比对，适用于无法改 nginx 配置的场景。

> 建议 nginx 与应用层配置一致；当服务直接暴露（无 nginx）时，应用层白名单是唯一防线，务必开启 `TRUST_PROXY=false` 或正确信任前置代理。

### 4.3 行为约定

- 白名单**全局生效**：代理路由与管理后台均受其约束。
- 未命中白名单 → 返回 `403 Forbidden`，不泄露后台地址或映射信息。
- 修改白名单后**热生效**，无需重启。

## 5. 访问日志

**目的**：记录每一次访问，便于审计、排查与流量分析。

### 5.1 记录字段

每条访问日志包含：

| 字段 | 说明 |
| --- | --- |
| `time` | 访问时间，本地时区 `YYYY-MM-DD HH:mm:ss.SSS` |
| `ip` | 客户端 IP（取 `X-Real-IP`/`X-Forwarded-For`，依赖 `TRUST_PROXY`） |
| `method` | 请求方法（GET/POST/…） |
| `path` | 请求路径（含 query string） |
| `status` | 响应状态码 |
| `elapsedMs` | 处理耗时（毫秒） |
| `mapping` | 命中的映射 `prefix`（未命中记 `-`） |
| `target` | 实际转发的目标 URL（仅代理请求） |
| `userAgent` | 客户端 User-Agent |
| `bytes` | 响应字节大小（可选） |
| `streamType` | 响应类型：`stream`（流式，SSE/chunked）/ `buffer`（非流式）/ `-`（非代理） |

### 5.2 存储位置

- 访问日志写入**独立的 JSONL 文件**（默认 `logs/access.jsonl`，每行一条 JSON），**不写入 `config.json`**。
- 理由：访问日志是高频追加数据，与低频读写的配置文件分离，避免撑大配置、便于轮转与清空。
- 支持按文件大小轮转（达到 `maxFileMB` 切分）与按天数清理（`retentionDays`），保留最近 `keepFiles` 个历史文件。

### 5.3 记录范围

- **管理后台自身的请求（`/{token}/` 下，含页面与后台 API）默认不记录**，避免后台轮询刷屏。
- 默认记录：代理转发请求，以及其它外部访问（如未命中前缀的 404）。
- 若想更严格、**仅记录代理转发**，可设 `accessLog.logProxyOnly = true`（此时连 404 等外部访问也不记录）。
- 取真实客户端 IP 依赖 `TRUST_PROXY`（经 nginx 反代时设为 `true`）。

### 5.4 查询与管理

后台「访问日志」页面支持**分页浏览**（默认每页 100 条）、按 IP / 路径 / 状态码 / 时间区间筛选、点击**「详情」**查看单条全部字段（含 User-Agent、响应字节等表格未展示项）、清空与导出。接口见 [API 文档 · 访问日志](04-API文档.md#六访问日志)，配置项见 [配置说明 · accessLog](03-配置说明.md#accesslog-访问日志)。

## 6. 连通性测试

后台提供对单条映射的连通性测试：选择一条映射，服务端以该映射的 `target` 发起一次探测请求（如 `HEAD` 或轻量 `GET`），返回状态码与耗时，帮助快速判断目标是否可达、目标是否需要额外鉴权头等。

详见 [API 文档 · 连通性测试](04-API文档.md#连通性测试)。
